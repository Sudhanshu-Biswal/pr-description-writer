# PR Description Writer

Write pull request descriptions that make reviewers happy — clear, structured,
and complete. Reads the actual git diff so you never have to explain your own
code twice.

---

## How to trigger

User runs:
```
/pr-description-writer
```

Or says anything like:
- "Write my PR description"
- "Help me open a PR"
- "What should I put in my pull request?"

---

## Step 1 — Gather context

Run these commands in order. Use whatever succeeds:

```bash
# 1. Get the diff (try each fallback in order)
git diff main
git diff master
git diff --staged
git diff HEAD~1

# 2. Get commit messages
git log main..HEAD --oneline
git log master..HEAD --oneline
git log HEAD~5..HEAD --oneline   # fallback

# 3. Get changed file list
git diff main --name-only
git diff --staged --name-only

# 4. Get current branch name
git branch --show-current

# 5. Check for open issue references in commits
git log main..HEAD --format="%B"
```

If all git commands fail, ask the user to paste the diff or describe what changed.

---

## Step 2 — Detect PR type

Classify the change automatically from diff + commits. Use this priority order:

| Signal | Type |
|---|---|
| `feat:` prefix OR new files added | ✨ Feature |
| `fix:` prefix OR bug-related keywords | 🐛 Bug Fix |
| `refactor:` OR no logic change, only restructure | ♻️ Refactor |
| `chore:`, `ci:`, config/workflow files only | 🔧 Chore / Config |
| `docs:` OR only `.md` files changed | 📝 Docs |
| `perf:` OR benchmark-related | ⚡ Performance |
| `test:` OR only test files changed | 🧪 Tests |
| Mixed / unclear | 🔀 Mixed Changes |

Use the emoji + label in the PR title prefix.

---

## Step 3 — Generate the PR title

Format:
```
<emoji> <type>: <concise present-tense summary> (#<issue> if found)
```

Rules:
- Max 72 characters
- Present tense, active voice ("Add rate limiting" not "Added rate limiting")
- No "WIP", no "Claude", no AI attribution
- Include issue number if found in commits (e.g. `fixes #123`, `closes #456`)

Examples:
```
✨ feat: Add rate limiting to public API endpoints (#234)
🐛 fix: Prevent null pointer on empty user profile load
♻️ refactor: Extract auth logic into reusable middleware
```

## Step 4 — Write the description

Use this exact markdown template. Fill every section. Never leave a section empty — if something doesn't apply, write one sentence saying why (e.g. "No breaking changes introduced.").

```markdown
## What changed
<!-- 2-4 sentences. What does this PR do? Why does it exist? -->

## Changes breakdown
<!-- Bullet list of specific changes. Group by area if large diff. -->
- 
- 

## How to test
<!-- Step-by-step. Be specific enough that someone unfamiliar can verify. -->
1. 
2. 

## Screenshots / output
<!-- If UI change: add before/after. If API change: show example response. If N/A, say so. -->

## Breaking changes
<!-- List anything that changes existing behavior, APIs, or contracts. -->
<!-- If none: "None." -->

## Related issues / links
<!-- Link issues, tickets, docs, or Slack threads. -->
<!-- If none: "None." -->
```

---

## Step 5 — Writing rules

Apply these to every word you write:

**Voice and tense**
- Active voice, present tense always
- "Adds X" not "X was added", "Fixes Y" not "Y has been fixed"

**Clarity**
- Front-load the most important thing in each sentence
- No hedging: cut "rather", "quite", "somewhat", "in order to", "the fact that"
- No filler phrases: cut "This PR...", "As part of this change...", "Going forward..."

**Tone**
- Write for the reviewer, not yourself
- Assume they haven't read your commits
- Help them understand in 30 seconds what changed and where to start reading

**Never include**
- AI attribution ("Generated by Claude", "Co-authored by AI")
- Filler like "I hope this helps"
- Obvious statements ("This changes the code")

## Step 6 — Output format

Print in this order:

```
**PR Title**
<generated title>

---

**PR Description**
<full markdown description>

---

**Quick copy**
> Paste title into GitHub PR title field, description into body field.
```

---

## Edge cases

**Huge diff (500+ lines)**
Summarize by file group, not line by line. Focus on intent, not implementation detail.

**No commits found / detached HEAD**
Fall back to `git diff HEAD~1`. If still empty, ask user to paste the diff.

**Monorepo**
Group changes by package/service in the breakdown section.

**No meaningful commit messages (e.g. "fix", "wip", "asdf")**
Ignore them. Derive everything from the diff itself.

**Draft PR**
Prefix title with `[WIP]` only if user explicitly says it's a draft.
